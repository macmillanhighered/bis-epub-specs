  • Do not include iBooks specific metadata <meta property="ibooks:specified-fonts">true</meta> in OPF.
  • Do not include “com.apple.ibooks.display-options.xml” in “META-INF” folder.

2.3.1. XHTML Content Documents

  • XHTML File encoding and extensions must be used. Files must be saved with UTF-8 encoding format (not UTF-8-BOM).
  • XHTML file size: Recommended maximum file size for separated XHTML is 300kb.
  • XHTML file break levels (Granularity): Unless otherwise specified, XHTML files for bodymatter (chapter content) should be broken down into A-heads. Part Title, Chapter Title, and A-heads, should all be separate XHTML files.
  • Naming conventions: Parts, Chapters, and A-heads may be called something different (unit, section, module, etc), but the smallest should be the A-head (refer to 3.10.2). Therefore, if a title has units and modules, and the smallest XHTML file is a module, the module should be marked up as an A-head and the units should be marked up as chapters. If it is unclear what constitutes an A-head, B-head, etc. for a particular title, please ask for clarification from the ML project manager requesting the EPUB as soon as possible prior to beginning the conversion.
  • In Part level books: If a chapter 0 or intro chapter is being used and that introductory chapter is not underneath another Part section, then the chapter must be at the topmost level. Chapter 0 should therefore be marked up at the part level, not at the chapter level. For example:
    • Chapter 0
    • Part 1
      • Chapter 1
  • The A-head level file should contain all of the constituent B-heads, C-heads, etc. that belong to that section. If B-head level or below are requested to be broken into separate XHTML pages, please direct immediately to Content Standards. It is crucial that this A-head level chunking is maintained for enhancement purposes. Be sure to follow the Accessibility 6.10 Headings guidelines and ML guidelines for 3.10 Hierarchy and Heading Structure when tagging the head levels within the XHTML files.
  • Frontmatter and backmatter sections (e.g., copyright page, preface, index, appendix, references) should only have one XHTML per section.

2.3.2. File Naming

The EPUB file should be named: EPUB3ISBN13_EPUB.epub

For example, if the book ISBN is 9781319011741 and the EPUB ISBN supplied is 9781319065478, then the EPUB file should be named 9781319065478_EPUB.epub and the accompanying front cover JPEG file should be named 9781319065478_FC.jpg.

For sample chapter EPUBs, insert 'sample_' after the ISBN in the file name, e.g. 9781319065478_sample_EPUB.epub.

The folder containing the EPUB content should be named OEBPS.

Files referenced by CSS stylesheet should be in the same folder as the stylesheet. This would include font folders and any images used by the stylesheet.

A folder may have the following structure:

  • styles
  • | -- fonts
  • | -- *.ttf
  • | -- *.otf
  • | -- images
  • | -- *.gif
  • | -- *.jpg
  • | -- *.css

The XHTML file naming should use the following naming convention:

  • <First 3 letters of author’s last name>_<Print book ISBN>_<chxx_xx>.xhtml. Example: mor_9781319011741_ch01_01.xhtml
  • Use the same naming convention for the NAV file, using "nav" in place of <chxx_xx>. Example: mor_9781319011741_nav.xhtml
  • For front and end matter, replace “FM” or “EM” respectively instead of “nav” / “chxx_xx”, followed by the section name. Example: mor_9781319011741_FM_dedication.xhtml.
  • Use the same naming convention for the CSS file. Example: mor_9781319011741.css
  • For EPUBs without a corresponding print book ISBN, the EPUB ISBN should be used instead in the above file naming conventions.
  • Use "toc.ncx" for the NCX file.

Filenames must be consistently named throughout the production process to preserve cross reference integrity. Since EPUBs may be used at any part of the production process for demos or active work, the filenames must remain the same.

2.3.3. Content Organization in EPUB

  • The EPUB should be concatenated and include all book content from beginning to end (including front cover, inside front cover, inside back cover, and back cover and any inserts) with the exception of blank pages, which should be removed.
  • Multi-column text should be converted to single column text within the EPUB.
  • Whenever possible, two-page spread designs should be converted and incorporated into the normal flow of the EPUB. There are exceptions, for example, infographics that are complex and span two pages. In this case, it is expected that the pages would be converted to a single image with long alt text added to describe the image in detail. Any questions or doubts, please contact Macmillan.

With Macmillan EPUB CSS, for content to be rendered in all-caps, it should be marked up with class="cap" . For content to be rendered in small-caps, it should be marked up with class="smallcap" .   • Key terms and definitions should be tagged using proper html element tags. See 4.3 Glossaries.
  • Learning Objectives should be tagged using the correct Semantic markup. See 4.6 Learning Objectives.

When in-chapter questions have numbered answers in and end of chapter section or in a backmatter Answers or Appendix section, follow the below standards:

    • For College titles (where audienceType=higher-ed in supplied metadata): The question number must have a cross-reference link to the corresponding numbered answer in its separate section.

    • For High School titles (where audienceType=schools in supplied metadata): Do not link the question to its answer. (Exception: If answers are included in footnotes, follow standard Note linking instructions. See 4.8 Notes.)

2.3.4.1. Element IDs

Every element in the HTML document within the body element should have an ID. This ID must be unique across all publications and cannot start with a number or special character.

ID’s must not change between iterative batches.

2.3.4.2. All Text Must be Available in a Logical Reading Order

All content must be presented in a logical reading order. Use semantic and structural markup to define the natural reading order of the primary narrative. Semantically distinguish secondary material such as footnotes, references, figures, tables, lists, and other auxiliary content from the primary narrative and identify their structure. Use HTML tags, not CSS, to position elements and determine reading order.

  • Secondary content is identifiable from logical reading order by inclusion in aside and figure tags
  • Keywords, equations, and figures should appear after the paragraphs in which they are mentioned. They should appear in the following precedence order:
    1. Keyword definitions
    2. Equations
    3. Figures
  • The order of the markup in content files matches the primary narrative (See 3.3 Content Order of Chapter Openers)
  • The publication is readable from beginning to end without scripting or styles applied.

http://kb.daisy.org/publishing/docs/html/order.html

2.3.4.3. Content Order of Chapter Openers

  • The content order of Chapter Opening pages that are designed across a spread should always be placed in the EPUB in the following order:
    • --Chapter Opener Number (including the word “Chapter”)
    • --Chapter Title
    • --Chapter graphic of whatever sort: it needs to be provided per the specs of graphics, as one piece with credit and alt text (contact Macmillan if you have any questions about the graphic treatment)
    • --Chapter Opener Outline
    • --Learning Objectives
    • --Opener text and/or first heading
  • Note: not all books will include all of the above features, such as chapter outline or learning objectives.

2.3.5. Cover

  • The EPUB requires a cover image linked in the manifest, e.g., <item id=“cover-image” href=“9781319123456_FC.jpg” media-type=“image/jpeg” properties=“cover-image”/> .
  • Front Cover (FC), Inside Front Cover (IFC), Inside Back Cover (IBC), and Back Cover (BC) pages should be separate XHTML files within the EPUB package, each tagged with the attribute epub:type=”cover” , and each having their abbreviation (FC, IFC, IBC, and BC) used as the page number in the navigation TOC.

    Note:

    • Alt-text is required for covers. The front cover must follow one of the below templates, where applicable:
      • Cover: [title] by [authors]
      • Cover: [title], [edition of title] by [authors]
      • Cover: [title], edited by [name/s of editors]
      • Cover: [title], [edition of title], edited by [name/s of editors]
    • Only if the image on the cover serves a purpose, follow with a brief description of the imagery. In most to all cases, there is no need for description, as the imagery on the cover is usually decorative.
    • If covers are not available at the time of EPUB conversion (e.g., with sample chapter EPUBs) a placeholder front cover page that reads "cover image to come" must be included.
  • Remove any barcodes and price listings from the front cover, inside front cover, inside back cover and back cover.
  • Front cover (FC) is the only cover that is mandatory. Inside front, inside back, and outside back covers are not required to be present. They may also be omitted upon request. Additionally, content in the covers (aside from the front cover) may be rendered in text, if it is requested and possible to do so. If so, the page can still be numbered appropriately, as above: IFC, IBC, BC. For instance, a brief TOC may appear on an inside front cover. If requested, the TOC may be rendered as text and the page number becomes IFC.
