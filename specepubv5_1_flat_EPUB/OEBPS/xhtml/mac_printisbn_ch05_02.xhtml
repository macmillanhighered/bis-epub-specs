When aside, article and nav elements with headings are included in a section, the heading rank MUST start one level lower than the containing section (e.g., if a section has an h3 heading, an aside within it will start with an h4 heading).

5.1.1. Deep Nesting

When the document hierarchy exceeds six levels of nesting, re-use the h6 element for the seventh nesting level and deeper. Avoid deep nesting of content, if possible, as the flattened structure below the sixth heading level impacts on the ability of users of assistive technologies (AT) to move through the document without repeatedly returning to the table of contents.

Do not use heading elements where the heading does not reflect the document hierarchy, as this use will impact on movement by AT users (e.g., inside figure captions).

Sectioning content should not have as its only direct descendent (child) element another sectioning content of the same type. For example, a section should not be the only child of a section, and an aside should not be the only child of an aside. Doing so is semantically unnecessary and serves to complicate the structure of the book. Note that this restriction does not apply to div. Please refer to the tagging guidelines 3.1 section 13, Untitled Sections.

5.1.2. Titled Sections

[HTML5] heading elements must be used to provide headings only for [HTML5] sectioning content (including the body element).

When using a heading element for a section, the rank must reflect the nesting of that section within the document hierarchy (as noted in 4.3.10 Headings and sections [HTML5]). For example, a top-level section will have an h1 heading, child sections of it will have h2 headings, and so on. Note that untitled sections are included when determining the current heading rank (see 3.10.3 Untitled Sections for more information).

The heading does not have to be the first element of the section, and may be included inside a [HTML5] header element. If the section also includes a subtitle in a separate element, both the heading and subtitle must be enclosed in a header.

          <section epub:type="chapter" class="chapter" id="fri_9781319065669_eVbIaagB1n" role="doc-chapter"> <header id="fri_9781319065669_LwWNoIjON9"> 	<h1 epub:type="title" class="title" id="fri_9781319065669_Dxi5OQToLl"><span epub:type="label" class="label" id="fri_9781319065669_6kr8vCzWUf">Chapter </span><span epub:type="ordinal" class="ordinal" id="fri_9781319065669_rWNSvdovxL">1 </span>The Battle</h1> 	<p epub:type="subtitle" class="subtitle" id="fri_9781319065669_LVU2ZxvHSQ" role="doc-subtitle">Once More Unto the Breach</p> </header> ... </section>

5.1.2.1. Hierarchy

For titles that have parts:

  • Title = <h1>
    • Part =  <h1>
      • Chapter =   <h2>
        • A-head =   <h3>
          • B-head =   <h4>

For titles that do not have parts:

  • Title = <h1>
    • Chapter =   <h1>
      • A-head =   <h2>
        • B-head =   <h3>

For the title of a book, use <h1> as long as the page has an epub:type attribute with value " titlepage". The title defined in the OPF is considered by reading systems as the title of the book.

When aside elements with headings are included in a section, the heading rank MUST start one level lower than the containing section (e.g., if a section has an h2 heading, an aside within it will start with an h3 heading)

  • Title = <h1>
    • Chapter =  <h1>
      • A-head = <h2>
        • Aside = <h3>

Subheadings should be marked up using a <p> with an epub:type attribute having the value " subtitle"  and an ARIA role attribute "doc-subtitle" and the heading and subheading should be wrapped inside a <header>.

5.1.3. Untitled Sections

If a section of content in the document hierarchy does not have an explicit heading (e.g., an untitled introduction) ‒ or sectioning is used to group significant content, such as learning objectives, that a user using an assistive technology would need access to ‒ the [ARIA] aria-label attribute MUST be used to provide an accessible name.

That name must be descriptive and intelligible to a reader. Adding this attribute allows users of assistive technologies to navigate to the section despite the lack of a heading, in addition to giving meaningful context.

The aria-label attribute must contain a value unique to the XHTML page it appears on. Otherwise it risks confusing a user of assistive technology if there are two or more sections with the same title.

Please note that even if there is an untitled section, the heading ranks must be sequential. Skipping heading rankings makes subsequent headings unreachable by assistive technology.

  • section
    • header
      • h1
    • section
      • header
        • h2
      • section (aria-label)
        • section
          • header
            • h3

5.1.5. CFI and Navigation Titling

Macmillan Learning uses Canonical Fragment Identifiers or CFIs to integrate readings from EPUBs into courses. The CFI identifier marks a specific point in an EPUB and allows us to create links (sometimes called locators) that go to that point or that open a snippet (a small portion of) the EPUB. CFIs are created based on logic of nesting and schema naming of <section> and <aside> elements.

The name for the CFI is generated on ingest to Frost and a new attribute is added to <section> elements and <asides>, data-cfititle. This is generated based on the value in <header> element and will pick up the value, casing and all.

In some cases, MPMs and Editors will request that this CFI be different from this logic in Casing or in Value. For example, an A-head does not include an ordinal, but it is required to in the navigation and CFI title, so the A-head name is "Drafting a Paper" but in the navigation and/or attribute data-cfititle, it is "1.1 Drafting a Paper". Therefore the h2.a-head or h3.a-head element is the "Drafting a Paper" and the parent <section> has an attribute data-cfititle="1.1 Drafting a Paper"

5.1.5.1. Adding data-cfititle attribute

  • data-cfititle attributes should be added only to <section> or <aside> elements.
  • If the content is already in Frost, the attribute will already be included, so the value can be updated directly in the code. If this request is done ahead of Frost ingestion then an attribute will need to be added

5.1.5.2. Conventions

  • Follow rules on casing. If content is in all capitals, use appropriate styling to obtain the all-caps or small-caps style and render the data-cfititle attribute value in title case.
  • Subtitles should be divided by a ":" (colon) between the main heading and subtitle
  • Special characters in attribute should be written as entities:
    1. double quotes (") should be &quot;
    2. ampersand (&) should be &amp;
    3. single quotes (') should be &apos;
    4. less than (<) should be &lt;
    5. greater than (>) should be &gt;
    6. Left single quotation mark (‘) should be &lsquo;
    7. Right single quotation mark (’) should be &rsquo;
    8. Left double quotation mark (“) should be &ldquo;
    9. Right double quotation mark (“) should be &rdquo;
  • If a heading contains MathML, follow the conventions for converting titles in navigation. Markup must not be included in the data-cfititle attribute value. Math must either be spelled out according to alt text or rendered as HTML entities, following the rules against using specific entities (for instance, vulgar fractions) which are enforced by Schematron.
