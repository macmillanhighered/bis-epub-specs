5.3.6.1. Packaging of Footnotes

  • Footnotes that use glyphs (e.g., asterisks, crosses, etc.) instead of numbers should be converted to numbered footnotes. If the book does not already have numbered footnotes, these should simply be numbered sequentially using arabic numerals (1, 2, 3...). If the book already has numbered footnotes with arabic numerals, the glyph footnotes should instead use lower-case roman numerals (i, ii, iii...) to differentiate the two sets of footnotes.
  • All endnotes, regardless of where they are located in the print book (bottom of page, end of chapter, end of book), must be gathered in one file and placed in the backmatter section of the EPUB. This includes all footnotes, whether they were originally numbered or had glyphs that were converted to numbers. The notes need to be contained in a stand-alone XHTML file. If the print book already has the Notes/References chapter in the backmatter, then the work is done and the vendor just needs to ensure this section is in a file on its own. If the notes/references are at the end of each chapter, or at the bottom of pages, these sections should be removed from within the chapter file, and combined into a new backmatter section. The vendor should create a Notes backmatter title and insert the section preceding the Index of the book. They should insert the appropriate chapter heads around each set of notes. Use the following layout for guidance.
  • All notes must also have back linking applied to them so that a user can go to the full notes section at the back of the book and click on the citation number to be taken to the location of the citation in text.
  • Organize footnotes and endnotes so that they do not interrupt the flow of the narrative but can be ignored or read as desired.
  • Notes within the primary narrative should be identified using the aside element with an appropriate epub:type value, such as footnote.
  • Groups of notes that appear together at the end of a section of the narrative do not need to be tagged as aside, but should be identified by an appropriate epub:type value. Groups should be presented in a list.
  • By default, footnotes are gathered with endnotes. However, footnotes may appear on the same XHTML page upon request. This is typically the case with books that have multiple readings.
  • For footnote references with class and EPUB type "noteref" and ARIA role "doc-noteref", which also appear on the same XHTML page as their related footnote, include an aria-describedby label equal to the value of the href.

Footnote:

            <p class="noindent"><a class="noteref" href="#fn0-1-2" id="fn0-1-2a">1</a>Modeling ideas and inference concepts that pertain to statistics also pertain to data science, which sits at the intersection of statistics, computer science, and application.</p>
  • Notes that appear within a table should follow the table or be grouped together with the table inside a figure tag. Avoid using tfoot, because that is intended for a table column summary, not for footnotes.
  • Note references should be tagged using the HTML5 a element the epub:type attribute with the value noteref.
  • It is required to back-link notes to the text referent for easy review of the related text.

5.3.7. Labeling Aside Elements with ARIA and Headings

There are two ways to label an aside element:

  • add an aria-label attribute with a unique value
  • add an aria-describedby attribute pointing to a heading descendant of the aside that is unique from the rest of the headings and aria-labels in the xhtml page

All aside labels must be unique, regardless of whether aria-describedby or aria-label is used.

If multiple asides have the same heading, do not use aria-describedby. Use aria-label, use the title in the aside, and enumerate the elements. For example,

          <aside class="tip" aria-label="Tip 1"> <header> <h3>Tip</h3> <header> ... <aside class="tip" aria-label="Tip 2"> <header> <h3>Tip</h3> <header>

When aria-labels are used, they must be intelligible and human-readable.

5.3.7.1. Incorrect examples

            <aside class="tip" aria-describedby="tip1"> <header> <h3 id="tip1">Tip</h3> <header> ... <aside class="tip" aria-describedby="tip2"> <header> <h3 id="tip2">Tip</h3> <header>

Even though the ID’s are unique, the string pointed to is not. Both aria-describedby attributes point to different ID’s of elements containing identical strings.

            <aside class="tip" aria-label="tip001"> <header> <h3>Tip</h3> <header> ... <aside class="tip" aria-label="tip002"> <header> <h3>Tip</h3> <header>

Even though the aria-labels are unique, these are not human-readable.
