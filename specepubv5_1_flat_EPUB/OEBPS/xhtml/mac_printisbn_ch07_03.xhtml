Structure tables so that they are accessible and so that the meaning of the information is retained (e.g., the relationships between rows and columns). Use headers properly to facilitate table navigation. If the textual contents of a table are required for comprehension of the document, use proper and complete markup for text and tabular data, including headers and scope attributes for tables. Include a caption to give context to the table. For complex tables, include a summary of the structure to guide the reader. Footnotes should appear with the table--wrap the table and footnotes in a figure tag.

  • Tables should not be used for presentational or fixed layout of content. For example, do not use tables to mimic the print placement of marginal annotations or multi-column content.
  • All table heading cells identified using th elements.
  • Tables should not be accompanied by <thead> or <tbody> elements. Instead, use <th> or the headers="" attribute. When more than one header applies to a column, attach the headers attribute to each cell to clarify which headers go with which cells. Click here for a sample of a complex table in HTML.
  • If a table is secondary content (not part of the logical reading order) enclose it with a figure tag.
  • Summaries are required for complex tables. Complex tables include: tables with colspan or rowspan, tables where one column has no data or some column has null value like ' - ', tables with exceptional size (6 or more columns and/or 6 or more rows), or tables not meant for navigation.
  • If a table is present only to inform the user of how a certain type of table is constructed, include a table summary that explains its pedagogical purpose.
  • If a table cell is multiple paragraphs long, all of the content inside of the cell must be wrapped in individual paragraph tags, not just after the first paragraph break.

  • Tables must not be set as images
  • Complex tables require summary
  • th element must not be empty. For empty cells, use td, even if within thead
  • scope attribute must be used on th elements to locate reader in tables. For this purpose, th is allowed to indicate beginning of row as well as the beginning of a column.

Tables as images

This request occurs during the following two scenarios:

  1. Annotated Tables (reading titles) captured as images to preserve the context on content/presentation
  2. Editorial preferences to retain standard table as images (title specific custom request from Editor/CPM)

Tables must never be set as images. If such a request is made, please forward it to Content Standards. If an image capture is decided to be pedagogically necessary, the image will require at least a table fallback or a table summary / extended description. The extended description must explain why the table has been set as an image. For example:

“A screenshot of a program output. The table in the screenshot...”

“An annotated table...”

Table Summary

Scenarios when a table summary is required:

  1. Complex tables
  2. Table is present only to inform user of how a certain type of table is made. Table must explain pedagogical purpose
  3. Table is captured as an image and table fallback

Summaries can be marked up like extended descriptions and will require extended description icons to be used.

Complex tables

Complex tables include: tables with colspan or rowspan, tables where one column has no data or some column has null value like '-', tables with exceptional size (6 or more columns and/or 6 or more rows), tables with multiple headers, or tables not meant for navigation.

Note: thead and tbody elements must be used to indicate the header and body of the table.

Oversized Tables

Overview

Oversized tables are tables with more than 1,800 cells or more than 20,000 characters in the HTML markup. Since the latter categorization is not possible to identify in the content stage, Content should focus on tables that are more than 1,800 cells. EPUB compositors can address both scenarios via Schematron validation (refer to specification status above).

When Content is able to identify an oversized table or one is reported to Content by the compositor based on Schematron validation, Content should give guidance based on pedagogy as to where it makes sense to split up the table. Content Standards will assist when it is requested.

Splitting a Table

Table headings must be maintained where necessary in all split tables. Every table must inform the user that it is split into multiple tables, even the first (original) table. If Content does not provide guidance on the naming convention for split tables, please use the following convention:

(x of y)

Where x indicates the current table’s order and y indicates the total number of tables. For example:

(1 of 3)

(2 of 3)

(3 of 3)

Other potential conventions could include alphanumeric ranges. For example:

  • “Glossary/Glosario A-F”
  • “Elements Hydrogen through Neon”
  • “Alkali Metals”, “Lanthanide Series”

Example

th Scope

th element has additional requirements:

  • Must not be empty
  • Must have scope attribute
    • if th does not have colspan or rowspan attr, scope must be equal to col or row
    • If th has colspan attribute, scope attribute must be equal colgroup
    • if th has rowspan attribute, scope attribute must be equal to rowgroup

Note: th element can be used to apply to row cells as well.
